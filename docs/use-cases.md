# 适用场景

## 适合

- 学习技术岗位加班证据如何结构化。
- 用 mock 数据理解证据链。
- 在内部讨论证据包字段和校验规则。
- 构建自己的本地脱敏流程。

## 不适合

- 上传真实聊天记录到公开仓库。
- 替代律师意见。
- 承诺协商、投诉、仲裁或诉讼结果。
- 处理未授权的他人隐私或商业秘密。

## 只有聊天记录，没有 Git

可以做：

- 建立聊天来源、行号、时间和 `quickLocator`。
- 形成 `wechat` 或 `group_chat` 证据项。
- 标记任务来源和处理结果。

风险：

- 缺少代码或系统侧结果互证。
- 可能只能达到 `medium` 或 `weak` 证据强度。

需要补充：

- 系统截图索引。
- 结果确认消息。
- 人工复核说明。

在本项目中可表示为：

- `sourceType: group_chat`
- `riskFlags: ["single_source_only"]`

## 有 Git，没有聊天记录

可以做：

- 记录 mock commit、时间、仓库和变更摘要。
- 形成 `git` 证据项。

风险：

- Git 只能说明代码活动，未必说明任务来源或加班安排。

需要补充：

- 任务来源说明。
- 发布窗口或结果确认。

在本项目中可表示为：

- `sourceType: git`
- `riskFlags: ["unclear_task_owner"]`

## 有发布夜记录

可以做：

- 把发布协调群聊、mock Git 和观察结果关联到同一事件。
- 使用 `release_night` 事件类型。

风险：

- 发布准备、发布观察和事后确认可能跨多个时间点，需要保守表述。

需要补充：

- 发布窗口。
- 观察结果。
- 回滚或修复记录。

## 有休息日任务

可以做：

- 使用 `rest_day_task` 事件类型。
- 记录任务来源、处理过程和结果确认。

风险：

- 休息日零散消息不一定构成完整任务证据。

需要补充：

- 明确任务要求。
- 处理结果或交付物索引。
- 人工复核动作。

## 证据不足，应排除

可以做：

- 将线索写入 `excludedCandidates`。
- 说明排除原因。

风险：

- 过度纳入会降低材料可信度。

需要补充：

- 更明确的时间、任务和结果证据。

## 何时应咨询律师

建议在以下情况咨询律师：

- 是否构成加班存在争议。
- 材料来源、隐私或商业秘密边界不清。
- 涉及复杂劳动合同、薪酬、岗位或管理制度。
- 需要判断正式维权路径。

本项目只做结构化和校验，不替代律师判断。
